## Slide 3 — 為什麼要做這個系統?

### 現況痛點

- ⏱ **24 小時內要給客戶報價**,但資訊散在 Email / Webex / Excel / 個人電腦

### 目標

> **集中平台 × AI 輔助 × 知識累積** → 讓團隊「接單更快、報得更準、學得更多」

## Slide 5.1 — 部署方式:獨立入口、共用底層

### 業務最關心的問題:「我要學兩套系統嗎?」

**答案:不用。** 使用者體感是兩個系統,但底層是共用的。

### 一張圖看懂

```
您看到的(使用者視角)
───────────────────────────────────────────
                                              
  輸入 cortex.foxlink.com.tw         輸入 quote.foxlink.com.tw
  登入 → 一般 AI 助理            登入 → 報價系統
  (聊天、技能、知識庫)         (冷色調、戰情室、成本分析)
                                              
          看起來是兩個系統,兩套畫面         
───────────────────────────────────────────

我們實際做的(技術底層)
───────────────────────────────────────────
                                              
   兩個網址 → 同一台後端伺服器
              (cortex-app)
                    ↓
              共用 ERP、AD、Oracle、KB
                                              
   網址不同 = cookie/權限/視覺全部獨立      
   但 RD 只要維護一套程式碼                  
───────────────────────────────────────────
```

### 對各位的意義

| 關心點 | 我們的回答 |
|--------|-----------|
| 登入要記兩組密碼嗎? | **不用**,同一個公司 AD 帳號,SSO 單一登入 |
| 兩邊切換麻煩嗎? | 另開分頁,兩個都能同時開。Cortex 也會加 "前往報價系統" 連結 |
| 萬一沒報價權限點到 quote 網址會怎樣? | 頁面顯示「您沒有報價系統權限,請聯絡系統管理員」,不會看到任何資料 |
| 為什麼要搞兩個網址? | 安全強化 — cookie / IP 白名單 / 二階驗證 都可以**只對 quote 網址從嚴**,不影響一般 Cortex 使用體驗 |
| 未來會有別的專業系統也這樣做嗎? | 會。這個架構未來可複製到 NPI、採購等模組 |

### 關鍵數字

- **UI 重做量**:相較「完全獨立開一套」省 ~50%(大約少 4,500 行程式碼)
- **維護成本**:長期 1 × vs 2 ×(Cortex 的升級自動帶到報價)
- **額外投入**:< 1 人天(Nginx 一個設定 + 前端一個切換器)

### 生命週期 8 階段

```

  ↓

  ↓

  ↓

  ↓

```

## Slide 8 — 業務 × PM 共治(關鍵設計)

### 我們的設計

```
業務(Sales)  ←─── 發起人 ───→    指派
                                    ↓
                                  PM
                                    ↓
                   業務 + PM ===共同管理===
                     ↓            ↓
                   邀請成員     管進度
                   踢除成員     改金額權限
                   結案         結案
                     ↓
        只有業務能「換 PM」(維護業務績效歸屬)
```

### 為什麼這樣設計

1. **業務是接單決定權持有者** → 擁有專案所有權 + 換 PM 權
2. **PM 是執行者** → 共享管理權,但不能撇開業務自己轉交
3. **兩人等權管成員** → 實務上 PM 忙於核算時,業務能獨立把工程加進來討論

**請業務部確認**:這個分工是否貼合實務?(例:目前業務和 PM 的關係是這樣嗎?)

- **高階主管不能改專案** — 要介入就正式加入為成員,維持責任鏈清楚

## Slide 10.1 — 戰情會議室 ★ 核心協作介面

### 為什麼需要「戰情會議室」而不是普通聊天室

24 小時 SLA 的壓力下,一個報價案常常**多人同時在跑不同環節**:業務追客戶、PM 跑核算、工程展 BOM、採購詢價、財務檢查… **如果還靠散落的 Email 和口頭同步,絕對來不及**。

### 戰情會議室解決的 3 件事

```
┌────────────────────────────────────────────────────────┐
│  ❶ 所有討論集中到一個畫面                               │
│                                                          │
│     任務進度 + 對話 + AI 建議 + 成員狀態                │
│     全部嵌在同一條訊息流,不用跳來跳去                  │
│                                                          │
│  ❷ 誰在做什麼、卡在哪,一眼看清                         │
│                                                          │
│     左欄任務樹 + 中央訊息流 + 右欄成員面板              │
│     紅(卡住)/黃(決策)/綠(進度)顏色一致            │
│                                                          │
│  ❸ 重要決策不會被錯過                                   │
│                                                          │
│     PM 做的關鍵決策自動釘選置頂,強制成員「已讀確認」    │
│     超過 30 分鐘沒確認?系統自動推 Webex 提醒           │
└────────────────────────────────────────────────────────┘
```

### 對比:傳統聊天 vs 戰情會議室

| 情境 | 傳統聊天 | 戰情會議室 |
|------|---------|-----------|
| PM 決定改用越南廠 | 聊天訊息混在一堆訊息裡,有人漏看 | **🟡 黃色「決策」訊息自動 Pin 置頂 + 要求已讀回執**,PM 看得到誰還沒讀 |

## Slide 12 — 金額欄位的特別保護

### 為什麼金額要特別處理?

> 客戶資訊(客戶名、料號、數量、交期)大家都看得到,本來就要傳閱討論。
> **真正需要嚴格保護的是價格類資訊** — 成本、毛利、合約單價、廠區內部時薪。

### 具體做法

| 對象 | 看到的樣子 |
|------|-----------|
| 業務 / PM | `$ 2.143 / pcs,毛利 16.6%` |
| 有價格權限的成員 | `$ 2.143 / pcs,毛利 16.6%` |
| **沒有價格權限的成員** | `Tier-A,毛利 Tier-M(中)` |
| 高階主管 | `$ 2.143 / pcs,毛利 16.6%` |
| 觀察者 | `Tier-A,毛利 Tier-M(中)` |

### 看金額必須「二階驗證」

```
點擊「成本」頁籤
      ↓
系統跳出:「請輸入動態密碼」
      ↓
手機 APP 跳出 6 位數 → 輸入
      ↓
驗證通過,10 分鐘內不會再問
```

> **為什麼加 2FA**:就算有人偷到你電腦(cookie),也看不到金額

## Slide 12.1 — 「資料加密會不會讓 AI 分析變笨?」★ 核心問題

### 會議中一定會有人問的問題

> 「我們把金額加密、設權限,那 AI 還能看到資料做分析嗎?
>  如果 AI 看不到,那我們還能用 AI 做歷史報價分析、定價建議嗎?」

### 答案:**完全不影響 AI 分析**

**原因**:加密擋的是「人和資料庫管理員偷看」,**不擋授權的 AI 程式**。

```
  ┌─ 人(業務、PM、甚至 DBA)───────┐
  │                                    │
  │   看到的 → 依權限 = 明文 or 代號   │
  │           (受加密保護)           │
  │                                    │
  └────────────────────────────────────┘
  
  ┌─ AI 分析程式(伺服器端自動跑)────┐
  │                                    │
  │   有合法的 KMS(金鑰)權限        │
  │   → 自動解密 → 跑分析             │
  │   → 結果再依使用者權限顯示        │
  │                                    │
  └────────────────────────────────────┘
```

### 三層 AI 分析架構

```
┌─────────────────────────────────────────────────────────┐
│ Layer 1 即時建議   < 2 秒                               │
│                                                           │
│   業務建新案 → AI 自動比對歷史脫敏案 → 秒回「類似 3 案」 │
│   對象:所有使用者                                       │
│   資料源:脫敏版公開知識庫                               │
│   安全:AI 看不到明文(KB 本來就是脫敏的)               │
├─────────────────────────────────────────────────────────┤
│ Layer 2 精確分析   5-30 秒                               │
│                                                           │
│   PM / Director 問「該客戶歷史贏單率」→ 跑 SQL + LLM     │
│   對象:業務 / PM / 高階主管                             │
│   資料源:完整資料(解密後)                             │
│   安全:Scrub 機制(見下一張)                           │
├─────────────────────────────────────────────────────────┤
│ Layer 3 統計儀表板   ★ 即時更新                          │
│                                                           │
│   主管開戰情室 → 即時看 SLA 燈號、贏單率、Delay 熱點    │
│   對象:高階主管 / admin                                 │
│   資料源:預先聚合的統計表(看不到單案明細)             │
│   安全:只出聚合數字,單案金額不暴露                     │
│   技術:資料一 commit 立即 refresh + WebSocket 推送      │
└─────────────────────────────────────────────────────────┘
```

### 關鍵結論

- 加密 **不會**讓 AI 變笨
- AI 看得到資料,**但用完就忘**(不訓練、不持久化)
- 使用者看得到什麼,**依然由權限決定**(Director 看明文,Observer 看 tier)

## Slide 12.2 — AI 外傳保護:Scrub 機制

### 還有一個細節業務可能擔心的

> 「Gemini / Azure AI 是外部服務,我們送資料過去,他們的 log 會不會留著我們的金額?」

### 真實情況

- 廠商承諾**不用我們的資料訓練模型**(合約有寫)
- 但**系統 log 可能留幾天**供 debug

### 我們的防線:Scrub(遮蔽)機制

```
送 AI 前:
  原始:「Apple 越南廠單價 $2.143,毛利 16.6%」
         ↓ Scrub(用代號換掉敏感值)
  送出:「[CUST_01] 越南廠單價 [PRICE_01],毛利 [MARGIN_01]」
         ↓ AI 分析完回來:
  回來:「[CUST_01] 越南廠單價 [PRICE_01] 對應市場區間合理,
         毛利 [MARGIN_01] 建議觀察未來 3 個月料價…」
         ↓ 我們這端再換回真實值
  顯示:「Apple 越南廠單價 $2.143 對應市場區間合理,
         毛利 16.6% 建議觀察未來 3 個月料價…」
```

### 結果

- **AI 廠商的 log 永遠只有 `[CUST_01]`、`[PRICE_01]`**,不含真實金額 / 客戶名
- 就算 log 外洩也無法反推
- 但 AI 的分析能力**一點都沒失**(它還是理解上下文)

### 業務可能還會問

**Q:那 AI 回來的答案是替回真值的嗎?**
A:是。回到我們系統後自動替回,使用者看到真實客戶名和金額(前提是他有權限)。

**Q:如果使用者沒權限看金額呢?**
A:系統會再做一次過濾,把真金額替成 Tier-A、Tier-M 這種代稱才顯示給他。**雙重保險**。

**Q:這樣會不會變慢?**
A:Scrub/Unscrub 都在毫秒內,使用者感覺不到。AI 本身耗時才是主要時間(5-30 秒不等)。

## Slide 20 — 風險提醒

### 我們必須誠實告訴各位的限制

#### ⚠ 風險 1:24 小時 SLA 是組織挑戰,不只系統

- 系統可以提醒、可以加速,但最終卡關常是**人要決策**
- **建議**:設「快速通道」— 高優先級案 24h,一般案 48h,避免每案都撞 SLA

#### ⚠ 風險 2:資訊外洩風險 — 人比系統脆弱

- 浮水印只能**追蹤**不能**防止**拍照
- **建議**:三地 admin 明訂**違規處理流程**(不是 IT 決定的)

#### ⚠ 風險 3:資料主檔維護責任

- 廠區費用主檔過期 = 核算不準 = 報價失真
- **建議**:財務承諾每季 review(超期系統會鎖住不讓核算)

#### ⚠ 風險 4:Oracle VPD(資料庫層權限)有 bug 風險

- 我們用**雙重防線**(應用層 + 資料庫層)降低風險
- 但技術上仍無法 100% 避免 silent leak
- **建議**:每季跑一次第三方 pen test(滲透測試)

## Slide 23 — Q&A

### 預期最常見問題(我們先準備的答案)

**Q1:業務現在沒在用 Cortex,會很難學嗎?**
A:Cortex 介面很像一般聊天軟體。我們會提供 1 小時操作培訓 + 說明書。

**Q2:為什麼要那麼多層安全?太麻煩了吧?**
A:金額是公司最敏感資訊之一。我們盡量讓**看資料順暢、只在關鍵動作**(進金額頁 / 匯出)加驗證,平常使用不影響。

**Q3:Pilot 失敗了怎麼辦?**
A:Pilot 環境獨立,不影響現有流程;失敗可回到現行 Excel 模式,資訊部修正後再試。

**Q4:我的部門沒人有 IT 背景,怎麼配合?**
A:每階段我們會派**對應聯絡人**駐點,不用各位學技術,只要回答業務問題。

**Q5:和現有 ERP / CRM 關係?**
A:
- ERP:抓 BOM + 客戶主檔(單向讀取)
- CRM:未來 Phase 3 雙向同步(Win/Loss 回寫)
- 新成本詢價系統:即時查詢採購價

**Q6:這個系統未來會開放給外部合作夥伴嗎?**
A:**不會**。這次規劃明確不做外部存取模組,僅內部使用。

**Q7:戰情會議室「強制已讀回執」業務文化能接受嗎?會不會太 pushy?**
A:理解這個顧慮。我們的設計是**只有 PM 主動標記「這是決策」的訊息才要求已讀**,一般聊天不需要。典型一個報價案 3-5 則決策訊息,不會造成轟炸。如果真的不適應,設定可以在 Phase 1 pilot 反饋後調整(例改為「30 min 後才推提醒」)。

**Q8:戰情會議室的「Delay 公開原因」會不會讓負責人有壓力?**
A:這是**刻意**設計。根據以往經驗,Delay 原因模糊化是 24h SLA 最大殺手。公開原因不是究責,是**讓團隊快速找人幫忙**。實務上我們看到:Delay 原因講出來,90% 的情況 1 小時內會有同事主動協助。

**Q9:加密後 AI 的回答會不會變奇怪?例如把數字變代號顯示給我?**
A:**不會**。AI 回答會自動「替回真實值」,您看到的是正常數字。代號只在 AI 外部服務的 log 裡,您永遠看不到。只有在**您沒權限看金額**時,系統會主動把真值換成 Tier-A 這種代稱,這是權限控管不是 AI 問題。

**Q10:AI 的歷史分析會看到我們過去的所有報價嗎?會不會洩到 Gemini/Azure?**
A:看分層:
- **Layer 1(即時建議)**:AI 看的是**脫敏版 KB**,本身就沒有真實金額
- **Layer 2(精確分析)**:AI 看解密後真實資料,但**送出前會 Scrub**(金額換代號),外部 LLM 的 log 只看得到 `[PRICE_01]`
- **Layer 3(儀表板)**:AI 完全不參與,純 SQL 聚合

**Q11:任務 Template 不夠用怎麼辦?**
A:兩種解法:
- 組員接到任務後**可自行增刪子任務**(不是綁死)
- 系統管理員可以**新增新 Template** 供未來使用(例:新事業處有獨特流程)

**Q12:如果業務和 PM 對同一件事有不同意見,以誰為準?**
A:系統**不做仲裁**。兩人都是管理者,各自能發「決策訊息」。若有衝突要業務流程處理(例:由事業處主管裁決)。但最常見情況是:業務決定「接不接這單、毛利底線」,PM 決定「怎麼做、分工」,實務上很少衝突。

## 附錄 C — 常見誤解澄清

### 「報價系統是不是要大家重學新系統?」
要學新介面,但**只有報價相關成員要學**。入口不同(`quote.foxlink.com.tw`),但介面邏輯和 Cortex 一致(同一個 RD 團隊做的)。一般員工不用管這個系統。

### 「金額加密會不會影響速度?」
影響輕微(每次查詢慢 5-15%,約毫秒級),使用者感覺不到。

### 「加密會不會讓 AI 變笨?」
**不會**。加密擋的是「非授權讀取」,AI 程式是授權的系統身份,有 KMS 解密權。使用者看資料受權限控管,AI 分析看全貌但外傳前先 Scrub。兩條獨立管道,互不干擾。詳見 Slide 12.1 / 12.2。

### 「AI 會不會取代 PM?」
不會。AI 做**核算**(純計算工作)+ **建議**(策略參考)+ **整理**(摘要寫作)。**決策仍是 PM + 業務主管**。

### 「戰情會議室是不是要大家整天盯著看?」
不用。重要事件會推 Webex,一般討論隨時進去看。就像 Cortex 既有的群聊,不是新的工作習慣。

### 「Template 是不是限制了我們的彈性?」
完全相反。Template 是**起點**不是終點,組員接到任務後可增、刪、改所有子任務。Template 只是避免每次都從零開始想「這次要拆什麼」。

### 「離職方案 B 如果 admin 不在會怎樣?」
- admin 是組的概念(3-5 人),不是單一員工
- 離職 webhook 發群組 mail + Webex 通知
- 超過 3 天未處理,系統自動升級到 CIO

### 「RAG 燈號和 AI 的 RAG 是同一個嗎?」
**不是**!這是投影片故意混淆的痛點。
- 本文件 RAG = Red/Amber/Green(紅黃綠燈號)
- AI 世界的 RAG = Retrieval-Augmented Generation(檢索增強生成)
- 兩個 RAG 都出現在 Cortex,請看上下文判斷

### 「獨立網址 quote.foxlink.com.tw 會不會要 IT 重新認證 SSL、設 DNS?」
是,但這是標配動作:
- IT 加一筆 DNS CNAME
- 加一張 SSL(自動續簽)
- Nginx 加一個 server block(5 分鐘)
- 都是一次性、< 1 人天工作
